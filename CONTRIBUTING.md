# Contributing

日本語版: [CONTRIBUTING.ja.md](CONTRIBUTING.ja.md)

Thank you for your interest! This applies to all repositories under
[@kitsuyui](https://github.com/kitsuyui) unless a repository has its own
CONTRIBUTING file. These projects are maintained by one person in personal
time — please be nice and respect the maintainer's time.

You can write issues and pull requests in **English or Japanese**, whichever
you prefer.

## Talk before you build (issue-first)

- **New features, breaking changes, large refactors**: open an issue first and
  wait for agreement before writing code. This protects your time, not just
  mine — I keep these libraries intentionally small, and many good ideas are
  better off living outside the core.
- **Typos and small, obvious bug fixes**: a direct pull request is fine.
  Please check existing issues first.
- **Not sure if it's a bug?** Open an issue and we can figure it out together.

### What will be declined

Stating this up front so nobody wastes effort:

- Unsolicited large refactors or style/grammar preference changes.
- Features outside the project's scope (each project tries to do one thing).
- Large pull requests that skipped the issue discussion above.
- Pull requests you do not plan to follow through on. Stale PRs cost review
  time with no outcome.

## What to expect from me

- I review and merge at my own pace. It may take days, sometimes weeks.
  A polite bump after about two weeks is welcome — needless bumping is not.
- Final decisions on merging rest with the maintainer. I will try to explain
  the reasoning when declining.
- If I closed something incorrectly, tell me why (politely). I make mistakes.

## Using AI

AI-assisted contributions are welcome within the rules in
[AI_POLICY.md](AI_POLICY.md). In short: you must understand and be able to
explain your contribution in your own words, and a human must be accountable
for it.

## Mechanics (kept short on purpose)

How to build and test differs per repository. The repository's README and
its CI configuration are the source of truth — run the same checks that CI
runs before submitting.

- Keep each PR focused on one change.
- Add or update tests when behavior changes.
- By submitting a contribution you agree to license it under the repository's
  license.

## Security issues

Do **not** open a public issue for vulnerabilities. See
[SECURITY.md](SECURITY.md).
